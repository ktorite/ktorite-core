# Ktorite

Backend framework built on Ktor and Exposed v1. Provides project scaffolding, database wiring, authentication, migrations, and an admin panel with minimal configuration.

## Install the CLI

```bash
curl -sSL https://raw.githubusercontent.com/ktorite/ktorite-cli/main/install.sh | sh
```

## Quick start

```bash
ktorite create-project myapp ./myapp
cd myapp
./gradlew run
```

The project comes with a `ktorite.properties` file in the project root:

```properties
# Ktorite configuration
# Change these for your environment (e.g. PostgreSQL)
secret=<auto-generated>
db.url=jdbc:h2:file:./data/db;DB_CLOSE_DELAY=-1
db.driver=org.h2.Driver
db.user=sa
db.password=
```

The session secret is auto-generated on project creation. Edit `db.*` values to switch to PostgreSQL. Then create a superuser and visit `/admin`:

```bash
./gradlew createsuperuser -Pargs="admin mypassword"
```

## Features

### CLI scaffolding
The `create-project` command generates a complete Gradle project with build files, database config, session auth, logback, and a sample model. The alpha `create-app` command (yet to be integrated completely) adds a new module inside an existing project.

### Database wiring
H2 file database for development (jdbc:h2:file:./data/db), PostgreSQL for production. HikariCP connection pooling with configurable pool size, timeout, and lifetime. The database instance is exposed via `config.db` for use in route handlers.

### Authentication
Two auth providers are bundled. Session auth with bcrypt password hashing, login and logout routes are auto-registered. Superuser-only access enforced by default. JWT auth with configurable issuer and realm. Both work alongside each other.

### Migrations
`SchemaUtils.createMissingTablesAndColumns()` runs on startup and auto-adds missing tables and columns without dropping existing data. For schema changes that need custom SQL, use `migration("name") { exec("ALTER TABLE ...") }`. Applied migrations are tracked in a `_ktorite_migrations` table and only run once.

### Security
CORS, HSTS, and CSRF configured through a single `security { }` block. CORS supports host allowlisting, custom methods and headers. HSTS with configurable max-age and subdomain preloading. CSRF with origin matching and custom header checks.

### Error handling
404s and unhandled exceptions return JSON responses (`{"error":"..."}`) instead of stack traces or Ktor's default HTML. In development mode, verbose error messages are included.

### Route helpers
`getB`, `postB`, `putB`, `deleteB` are trailing slash aware versions of the standard Ktor route functions. They match both `/path` and `/path/` so you don't need to worry about slashes in client requests.

### Admin panel
A separate [ktorite-admin](https://github.com/ktorite/ktorite-admin) component, bundled transitively. Provides CRUD for all registered models - login form, session auth, CSRF protection, paginated list views, detail view, and mass-assignment protection. Uses `Thymeleaf` templates. Add `ktorite-core` as a dependency and it is included in core.

## Usage

```kotlin
import java.util.Properties
import java.io.File

fun main() {
    val props = File("ktorite.properties").inputStream().use { Properties().apply { load(it) } }

    Ktorite.start {
        port = 8080
        developmentMode = true
        database {
            url = props.getProperty("db.url")
            driver = props.getProperty("db.driver")
            username = props.getProperty("db.user")
            password = props.getProperty("db.password")
        }
        auth {
            session { secret = props.getProperty("secret") }
        }
        registerModels(Product, Category)
        routing {
            get("/hello") { call.respondText("world") }
        }
    }
}

object Product : Table("products") {
    val id = integer("id").autoIncrement()
    val name = varchar("name", 128)
    val price = double("price")
    override val primaryKey = PrimaryKey(id)
}
```

## Performance

12-core machine, PostgreSQL. Same schema and data.

| Endpoint | Ktorite | Django (WSGI 24 workers) |
|---|---|---|
| `GET /`  | 25,900 RPS | 3,600 RPS |
| `GET /api/books` | 18,300 RPS | 3,500 RPS |

Benchmark runs with `hey` (100,000 requests, 200 concurrent connections, after JIT warmup):


JVM JIT + Netty event loop vs Python GIL + sync workers. Gap narrows with heavier logic. Django has 20+ years of ecosystem Ktorite cannot match at initial stage.

## How it compares to Django

| Area | Ktorite | Django |
|---|---|---|
| Throughput | 4-16x faster | Slower |
| ORM | Thin (Exposed, JDBC) | Feature-rich (relations, signals, expressions) |
| Admin | Basic CRUD (separate component) | Full-featured (search, filters, permissions) |
| Auth | Session + JWT, bcrypt | Built-in users, groups, permissions, OAuth |
| Migrations | Manual SQL via `migration()` | Auto-detected, reversible, dependency-tracked |
| Ecosystem | :( | 20+ years |
| Docs | Soon | docs.djangoproject.com |

Ktorite is early-stage. Feature depth and ecosystem will grow over time.

## Installation

### JitPack

```kotlin
repositories {
    mavenCentral()
    maven("https://jitpack.io")
}
dependencies {
    implementation("com.github.ktorite:ktorite-core:v1.0.0")
}
```

### Local build

```bash
cd ktorite-core
./gradlew publishToMavenLocal
```

## Project layout

```
ktorite-core/     - framework
ktorite-admin/    - admin panel (bundled transitively)
ktorite-cli/      - CLI tool
```
